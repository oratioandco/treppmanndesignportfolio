Prototyping in Code: How I Collapsed the Design-to-Ship Pipeline

Head of Product Design

Traditional handoff assumes designers and engineers work sequentially. We collapsed the pipeline: I prototyped in code with real APIs using Claude Code, and engineering consumed the prototype directly. No Figma specs. No redlines. Working code.

✱
The prototype is not a mockup. It is the first working version of the product.

Rethinking what a design prototype should be

§ 01

The Old Workflow

The traditional handoff looks like this:

1. Designer researches and designs in Figma 2. Designer creates specs and annotations 3. Designer redlines measurements and interactions 4. Engineering implements from specs 5. Design reviews implementation, finds mismatches 6. Back and forth for weeks

Every step introduces ambiguity. Every handoff loses information. And the prototype code that engineering writes bears no relation to the design thinking that produced it.

§ 02

The New Workflow

The pipeline I built for Bibel TV:

1. I design conceptually in Figma (layout, flow, hierarchy) 2. I prototype functionally in code using Claude Code + Cursor — real APIs, real data, real interactions 3. I iterate on the prototype with engineering feedback in real time 4. The prototype becomes part of the production codebase (component structures, API calls, data fetching patterns) 5. Engineering refines, not re-implements

The key difference: there is no handoff. There is continuous collaboration. The design artifact is working code, not a static spec.

§ 03

How Agentic Engineering Works in Practice

  1. 01

    Design Intent

    I start with conceptual design in Figma — layout, flow, visual hierarchy. This defines the what, not the how.

  2. 02

    Agentic Prototyping

    I use Claude Code and Cursor to prototype with real Bibel TV APIs. I generate component structures, data fetching patterns, navigation flows. I make design decisions in code — adjusting layouts, refining interactions, testing edge cases.

  3. 03

    Real Data Validation

    The prototype uses actual Bible TV content — real videos, real metadata, real user flows. This catches design problems that placeholder data never surfaces.

  4. 04

    Engineering Integration

    The engineering team reviews the prototype code. They extract component patterns, API structures, and navigation logic directly. Refinement, not re-implementation.

  5. 05

    Continuous Iteration

    When engineering finds a technical constraint, I iterate on the prototype in code. No Figma → spec → handoff cycle. The design evolves in the same medium as the product.

§ 04

Why This Is Better Than Figma Handoff

Speed. One iteration cycle that used to take 3-5 days now takes hours. I can adjust a layout, run it with real data, and evaluate the change immediately.

Clarity. Working code is unambiguous. A Figma spec says 'the card should feel premium.' Code says 'the card uses 16px padding, 4.5:1 contrast, and a 12px margin.' Engineers don't have to guess.

Shared language. When design and engineering both work in the prototype, they share a vocabulary. No more 'what the designer meant' — it's in the code.

Quality. The prototype is tested on the actual target platform (Android). We caught gradient rendering issues before a single line of production code was written.

§ 05

The Android Gradient Problem

A concrete example: I built the web prototype first with beautiful gradients. They looked great. When I moved to Android, the gradient rendering raised quality concerns — potential banding, performance questions on lower-end devices.

In a traditional workflow, I would have to file a ticket, explain the issue to engineering, wait for their assessment, maybe redesign. Instead: I iterated on the Android prototype in code, tested alternatives with real rendering, and found a solution that worked. The engineering team had a working reference, not a problem report.

§ 06

What This Means for Design Leadership

This workflow requires a design leader who is comfortable in code — not writing production systems, but building prototypes that demonstrate intent. It also requires an engineering culture that values prototype code as more than throwaway work.

I spent part of my role actively evangelizing this approach with the broader team. The shift is cultural: prototype code is a design artifact as valuable as a Figma file. When engineers accept that, the collaboration fundamentally changes.

§ 07

§ 08

What I Learned

The prototype is the best design spec because it eliminates ambiguity. Working code is unambiguous in ways that static specs never are. When engineering reviews code, they see exactly what works — and exactly what doesn't.

The cultural shift is harder than the technical one. Getting engineers to value prototype code requires trust and demonstration. Once the first prototype becomes production code, the shift happens organically. People stop asking 'is this ready for handoff?' and start asking 'what part of the prototype can I reuse?'

WHY THIS MATTERS FOR YOUR TEAM

If you're a design leader working with engineering teams, collapsing the handoff is the single highest-leverage process change available. It doesn't require new tools — it requires a different relationship between design intent and implementation. I used Claude Code to accelerate prototyping, but the insight is tool-agnostic: when your prototype is working code, the design-to-ship pipeline disappears.
